# Лабораторная работа №7. Шаблонизация

## Тема проекта: Трекер привычек

**Модель данных (Habit):**

1. `id` (string) — уникальный идентификатор записи.
2. `name` (string) — название привычки.
3. `description` (text) — подробное описание или мотивация.
4. `start_date` (date) — дата начала трекинга.
5. `category` (enum) — категория (Здоровье, Обучение, Спорт, Финансы).
6. `is_daily` (boolean/checkbox) — требуется ли ежедневное выполнение (Да/Нет).
7. `created_at` (date) — системная дата создания записи.

## Цель работы

Освоить принципы шаблонизации в PHP, как с использованием нативных PHP-шаблонов, так и с применением готового шаблонизатора (Twig). Улучшить структуру проекта, разделив логику обработки данных и представление.

## 1. Новая структура проекта (MVC подход)

Код проекта был разделен на логику (`src/`) и представление (`templates/`).
Также добавлен Composer для установки зависимости `twig/twig`.

**Файловая структура:**

```text
Lab7/

│   │   ├── layout.php          # Общий макет (header, footer, переключатель движков)
│   │   ├── form.php            # Шаблон формы добавления
```

## 2. Реализация шаблонизации

В проекте реализовано два движка шаблонизации. Переключение между ними происходит через GET-параметр `?engine=native` или `?engine=twig` (по умолчанию `native`).

### Шаг 1: Нативные PHP шаблоны
В `templates/native/layout.php` используется `include 'form.php';` и `include 'list.php';` для сборки страницы. В шаблонах используется конструкция `<?= htmlspecialchars($var) ?>` для безопасного вывода переменных, чтобы предотвратить XSS уязвимости. Логика полностью вынесена в `index.php`.

### Шаг 2: Использование шаблонизатора Twig
Через Composer установлен пакет Twig (`composer require twig/twig`).
Шаблоны переписаны с использованием синтаксиса Twig:
- В `layout.twig` используется `{% block content %}{% endblock %}` для обозначения контентной области.
- В `index.twig` используется `{% extends "layout.twig" %}` для наследования каркаса и `{% include 'form.twig' %}` для подключения фрагментов страницы.
- Переменные выводятся безопасно и лаконично через `{{ var }}`.

### Шаг 3: Дополнительная функция (Собственный фильтр Twig)
Был создан кастомный фильтр Twig `category_icon`, который принимает название категории (Здоровье, Обучение, Спорт, Финансы) и возвращает её вместе с соответствующим тематическим эмодзи.

**PHP-код (в `index.php`):**
```php
$categoryIconFilter = new \Twig\TwigFilter('category_icon', function ($category) {
    $icon = '📌';
    if ($category === 'Здоровье') $icon = '🏥';
    elseif ($category === 'Обучение') $icon = '📚';
    elseif ($category === 'Спорт') $icon = '🏋️';
    elseif ($category === 'Финансы') $icon = '💰';
    return $icon . ' ' . $category;
});
$twig->addFilter($categoryIconFilter);
```

**Использование в шаблоне (`list.twig`):**
```twig
<td>{{ habit.category | category_icon }}</td>
```

## 3. Контрольные вопросы

### 1. В чём отличие нативных PHP-шаблонов от шаблонизатора Twig? Какие преимущества и недостатки у каждого подхода?

**Нативные PHP-шаблоны:**
- *Преимущества:* Не требуют установки сторонних библиотек, работают предельно быстро (отсутствует этап компиляции), доступны все встроенные возможности PHP.
- *Недостатки:* Синтаксис вывода (например, `<?= htmlspecialchars($var) ?>`) громоздкий; легко забыть про экранирование, что ведет к уязвимостям XSS; сложнее реализовать полноценное наследование макетов; код быстрее становится трудночитаемым из-за смеси HTML и PHP-тегов.

**Шаблонизатор Twig:**
- *Преимущества:* Очень лаконичный и читаемый синтаксис (`{{ var }}` вместо вызова функций экранирования); автоматическое безопасное экранирование данных из коробки; мощная и удобная система наследования шаблонов (`extends`, `block`); наличие множества встроенных фильтров и циклов; не позволяет писать сложную бизнес-логику внутри шаблона (ограничивает выполнение произвольного PHP).
- *Недостатки:* Требует использования Composer для установки; есть небольшие накладные расходы на первую компиляцию шаблона (компенсируется кэшированием в production); необходимо изучение синтаксиса самого Twig.

### 2. Зачем разделять логику и представление в проекте? Какие проблемы могут возникнуть, если смешивать их в одном файле?

Разделение логики (подготовка данных, запись в БД, валидация) и представления (генерация HTML) — это базовая концепция паттерна MVC. Это необходимо для чистоты, переиспользуемости и масштабируемости кода.

Если смешивать всё в одном файле:
- Код превращается в нечитаемую «кашу» (спагетти-код), в которой сложно ориентироваться.
- Внесение изменений в дизайн или верстку может случайно сломать бизнес-логику (и наоборот).
- Разработка в команде усложняется: верстальщик не может спокойно работать с HTML-кодом, перегруженным PHP-логикой, а программисту мешает массивный HTML.
- Невозможно независимо тестировать код логики.

### 3. Что такое наследование шаблонов в Twig? Как работают `{% extends %}` и `{% block %}`?

Наследование шаблонов в Twig позволяет создать один базовый «скелет» (layout), который определяет общую структуру HTML-страницы (header, footer, навигацию и подключение стилей), а остальные страницы будут лишь расширять его, заполняя уникальным контентом.
- `{% block block_name %}` в базовом шаблоне (`layout.twig`) объявляет пустую область или область по умолчанию, куда дочерние шаблоны могут вставлять свой контент.
- `{% extends "layout.twig" %}` в самом начале дочернего шаблона (например, `index.twig`) указывает, что он использует `layout.twig` в качестве основы. Внутри дочернего шаблона описывается только контент для соответствующих блоков (например, `{% block content %}...{% endblock %}`), который заменяет одноимённые блоки в родителе.
