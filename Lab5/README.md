# Лабораторная работа №5. Объектно-ориентированное программирование в PHP

## Цель работы

Освоить основы объектно-ориентированного программирования в PHP на практике. Научиться создавать собственные классы, использовать инкапсуляцию, разделять ответственность между классами, а также применять интерфейсы для построения гибкой архитектуры приложения.

## 1. Архитектура и структура проекта

Для соблюдения стандартов профессиональной разработки проект разделен на независимые модули (классы). Везде используется строгая типизация (`declare(strict_types=1)`).

**Файловая структура:**

```text
Lab5/
├── style.css                        # Визуальное оформление (отделено от логики)
├── TransactionStorageInterface.php  # Интерфейс (контракт) хранилища
├── Transaction.php                  # Класс-сущность (DTO) транзакции
├── TransactionRepository.php        # Класс-репозиторий для хранения данных
├── TransactionManager.php           # Класс бизнес-логики (вычисления, фильтрация)
├── TransactionTableRenderer.php     # Класс (final) для генерации HTML-представления
└── index.php                        # Точка входа (сборка модулей и вывод на экран)
```

## 2. Реализация требований

1. **Инкапсуляция (Transaction):** Все свойства транзакции скрыты (`private`). Доступ к ним осуществляется только через геттеры. Также реализован метод `getDaysSinceTransaction()` для вычисления разницы в днях с текущей датой.
2. **Разделение ответственности:** 
   - `TransactionRepository` отвечает исключительно за хранение в памяти (массив скрыт).
   - `TransactionManager` выполняет математические вычисления (подсчет суммы, фильтрация, сортировка), не храня в себе сами транзакции.
   - `TransactionTableRenderer` занимается только генерацией HTML-кода таблицы.
3. **Полиморфизм и интерфейсы:** Внедрена зависимость через `TransactionStorageInterface`. `TransactionManager` получает интерфейс через конструктор, что позволяет в будущем легко подменить хранилище (например, на базу данных) без изменения кода самого менеджера.
4. **Документация:** Весь код прокомментирован с использованием стандартов PHPDoc (`@param`, `@return`).

В файле `index.php` создано 10 различных транзакций для демонстрации работы системы. На экран выводится таблица с отсортированными данными и небольшая статистика по операциям.

## 3. Контрольные вопросы

**1. Зачем нужна строгая типизация в PHP и как она помогает при разработке?**
Строгая типизация (`declare(strict_types=1);`) заставляет PHP четко следить за тем, какие типы данных передаются в функции и возвращаются из них. Это защищает от скрытых логических ошибок (например, когда строка неявно приводится к числу), делает поведение программы предсказуемым и заставляет писать более надежный код.

**2. Что такое класс в объектно-ориентированном программировании и какие основные компоненты класса вы знаете?**
Класс — это «шаблон», по которому создаются конкретные объекты. Основные компоненты:
- **Свойства (Properties):** Переменные внутри класса, которые хранят его состояние (например, ID или сумма транзакции).
- **Методы (Methods):** Функции внутри класса, определяющие его поведение (например, расчет дней).
- **Конструктор:** Метод `__construct()`, который автоматически вызывается при создании объекта для задания начальных параметров.
- **Модификаторы доступа:** `public`, `private`, `protected`, реализующие механизм инкапсуляции.

**3. Объясните, что такое полиморфизм и как он может быть реализован в PHP.**
Полиморфизм — это способность программы обрабатывать объекты разных классов одинаковым образом, если у них есть общий контракт (набор методов). В PHP это реализуется через интерфейсы или наследование, позволяя взаимозаменяемо использовать разные классы.

**4. Что такое интерфейс в PHP и как он отличается от абстрактного класса?**
Интерфейс — это строгий контракт, который содержит только сигнатуры методов без их реализации. Обычный класс может реализовывать несколько интерфейсов одновременно. 
Абстрактный класс — это частично готовый класс, который может содержать как абстрактные (пустые), так и уже реализованные методы, а также свойства. Наследоваться в PHP можно только от одного класса.

**5. Какие преимущества дает использование интерфейсов при проектировании архитектуры приложения? Объясните на примере данной лабораторной работы.**
Интерфейсы делают архитектуру гибкой и слабо связанной. В нашей работе `TransactionManager` опирается на абстракцию `TransactionStorageInterface`, а не на конкретный класс `TransactionRepository`. Это значит, что если мы захотим хранить транзакции в MySQL или Redis, нам достаточно будет создать новый класс (например, `DatabaseRepository`), реализующий этот же интерфейс. При этом код класса `TransactionManager` менять вообще не придется.
